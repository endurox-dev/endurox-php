PHP Endurox

Function API

NOTE: At this point in this products development cycle, it is assumed users have some understanding of basic Endurox client development. Many of these functions have a strong relationship to the original BEA Endurox API function. I recommend you where necessary refer to this documentation for additional information. BEA has their documentation online at:http://edocs.bea.com/endurox/ndrx65/index.htm. The Reference Material and Bookshelf are particularly good references.

Function Description Version Added Status
ndrx_array2ubf Move array values into UBF 0.2.0 Tested
ndrx_array2ubf (buffer #, array)
buffer#: The Endurox buffer reference number for values to be stored in. Must be a valid UBF/UBF32 Endurox buffer.
array: a 1 or 2 dimensional array to move the values from. The first dimension key should be the UBF/UBF32 Field names (a string) according to the Field Definition file. The second dimmension key (if present) will be taken as the FIELDID occurenence values, starting with 0. Skipped occurences result in a NULL UBF field id.

Returns TRUE/FALSE

Stores the values from the provided array into the destination UBF/UBF32 buffer. Values are placed into the UBF buffer with the Fchg Endurox API function. See the examples for a simple usage.

This function has no single equivalent Endurox API function.
See Also: ndrx_ubf2array, BEA Documentation of Fchg and related functions.
ndrx_ubf2array Move UBF values into an array 0.2.0 Tested
ndrx_ubf2array (buffer #, flag)
buffer#: The Endurox buffer reference number for values to taken from. Must be a valid UBF/UBF32 Endurox buffer.

flag: to indicate whether the primary index should be int or string They are set in a sequence of bits anded together.

0 0 0 = Perform no index translation, multi-occurrence.
0 0 1 = Store keys as the Field ID, coded same as 0.
0 1 0 = Store keys as the Endurox Field Number value.
1 0 0 = Store keys as the Field Name (String) value.

Returns a new 1 or 2 dimensional array with values populated from the UBF/UBF32 buffer. The first dimension key should be the Field Number or UBF/UBF32 Field names (a string) according to the Field Definition file. The second dimmension key (if present) will be taken as the FIELDID occurenence values, starting with 0. Skipped occurences result in a NULL value.

This function has no single equivalent Endurox API function.
See Also: ndrx_array2ubf, BEA Documentation of Fnext and related functions.

Low Level UBF/UBF32 Functions
Most of these UBF functions have one-for-one equivalents to the raw Endurox UBF functions. Where possible, the correct UBF/UBF32 calls are detected; however on some calls this is not possible so you must provide a type argument.

The list of these functions will likely grow in future releases. I tried to pick out the functions that were either very commonly used and/or with no equivalent method already created. If you find a specific function not listed here is needed, they can be created easily. Send me an email.

NOTE: As of version 0.4.0, the following functions (ndrx_fdelall, ndrx_foccur, ndrx_fdel, ndrx_flen, ndrx_fpres, ndrx_fadd, and ndrx_fget) can take a Field Name string in place of the Field ID parameter. If the provided argument is a string (as defined in PHP) the fldid function will be called to convert the string to its Field ID before processing.

ndrx_fadd Add new field occurrence. 0.3.0 Tested
ndrx_fadd (buff#, Field_ID, data, occur)
buff#: The buffer reference number for the data value to be added.

field_id: If numeric, the numerical field id for data to be added. If string, the Field Name for the corresponding Field ID. The conversion is automatic.

data: Data value to be added. The type of data is determined by the Field_ID type given. The length of the data value is automatically determined by the type and PHP.

occur: An optional field to allow the specification of a Field_ID specific occurrence. If the value is not specified, -1 is assumed which means to add the data as the next occurrence.

Return value is a long.

ndrx_fadd and ndrx_fchg are aliased to the same function code. The only difference between the two original functions is Fchg allowed the specification of an occurrence where Fadd did not.

See BEA documentation for Fadd and Fchg for more information.
